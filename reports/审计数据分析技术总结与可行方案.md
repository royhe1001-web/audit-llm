### 模块一：数据采集与预处理

**来源**：第 04 章 PPT + `overview_pandas_cleaning` / `Merge_data` / `Transform_data` / `Data_scaling` / `financial_imputation_tutorial`

#### 掌握要点

| 技术 | 核心方法 | 应用场景 |
|---|---|---|
| **源数据采集** | 直接复制、ODBC接口、数据库导出、审计软件模板 | 从被审计单位信息系统获取原始数据 |
| **审前调查** | 组织调查→计算机系统调查→数据库分析→数据需求说明书 | 明确数据来源、格式、范围 |
| **数据清洗** | `isnull()`/`dropna()`/`fillna()`/`duplicated()`/IQR异常值检测 | 处理缺失值、重复值、异常值 |
| **数据转换** | 格式转换、字段类型调整、字段名中文化、关系表合并 | 将原始数据转为可分析格式 |
| **数据合并** | `pd.merge()` 四种连接（inner/left/right/outer） | 多表关联、跨系统数据整合 |
| **缺失值填补** | SimpleImputer(mean/median/mode)、KNNImputer、IterativeImputer | 财务数据缺值处理 |
| **数据验证** | 记录数核对、金额合计验证、字段完整性检查 | 确保数据完整可靠 |
| **数据缩放** | MinMaxScaler、StandardScaler、RobustScaler、PowerTransformer、QuantileTransformer、Winsorize | 机器学习前的特征标准化 |

#### 关键认知
- **审计中间表**是本课程的核心概念——面向审计分析的数据存储模式，是对历史数据的选择和整合，**不应增加修改历史数据**
- 数据质量通过转换→清理→验证三步实现，是后续分析可信度的基础
- 不同缩放方法适用于不同数据分布：MinMax→有界数据，Z-score→正态分布，Robust→含异常值

### 模块二：数据库与 SQL

**来源**：`SQL_teaching` / `SQL_case` / `SQL_case-a` + 第 01/05 章 PPT

#### 掌握要点

| 技术 | 核心内容 | 应用场景 |
|---|---|---|
| **关系数据库基础** | 表结构、主键、外键、数据字典 | 理解被审计单位数据存储方式 |
| **DDL** | CREATE TABLE、字段类型定义、约束 | 建表、理解表间关系 |
| **DML** | INSERT、UPDATE、DELETE | 数据操作 |
| **DQL** | SELECT、WHERE、JOIN、GROUP BY、ORDER BY、HAVING、聚合函数 | 查询筛选、汇总分析 |
| **Python 连接** | pymysql + SQLAlchemy + `pd.read_sql()` | 将数据库查询结果导入 Python 分析 |

#### 关键认知
- 审计数据分析三种类型：**查询型**（SQL筛选排序）→ **验证型**（假设检验）→ **挖掘型**（聚类、异常检测）
- 查询型和验证型是目前应用最普遍的，但挖掘型在面对海量数据、经验有限时更具优势
- SQL 是审计人员的基础能力——被审计单位大多使用 SQL Server / Oracle / MySQL

### 模块四：监督学习——分类

**来源**：`ML_Classification_Tutorial` / `Payroll_Classification_Exercise` / `financial_classification_teaching_demo` / `unbalanced data` 系列

#### 掌握要点

| 算法 | 核心特点 | 审计应用 |
|---|---|---|
| **逻辑回归** | 线性分类器，可解释性强，输出概率 | 财务风险评分、欺诈概率预测 |
| **K-近邻(KNN)** | 基于距离，无需训练，对异常值敏感 | 相似交易查找 |
| **决策树** | 可解释，易过拟合 | 规则挖掘、审计决策路径 |
| **随机森林** | 集成学习，抗过拟合，输出特征重要性 | **审计首选**：稳健、可解释、能排序特征 |
| **梯度提升(GBDT/XGBoost)** | 高精度，参数敏感 | 复杂欺诈检测 |

#### 关键认知
- **不平衡数据处理是审计分类的核心挑战**：欺诈/违规比例通常 < 5%
- 准确率(Accuracy)在不平衡场景下会误导——98% 准确率可能只是把所有样本判为正常
- **应该关注 Recall（查全率）+ F1-Score + PR曲线**，而非单纯的 Accuracy
- 三种处理策略：欠采样（丢弃多数类信息）→ SMOTE过采样（合成少数类）→ 类别权重（class_weight='balanced'）
- **SMOTE + 类别权重组合通常效果最佳**
- 特征重要性分析可帮助审计人员理解哪些指标最关键（revenue_growth、debt_ratio 通常排名靠前）

### 模块五：监督学习——回归

**来源**：`ML_Regression_Tutorial`

#### 掌握要点

| 算法 | 核心特点 | 审计应用 |
|---|---|---|
| **线性回归** | 简单可解释，假设线性关系 | 基准预测、趋势分析 |
| **决策树回归** | 处理非线性，易过拟合 | 快速探索 |
| **随机森林回归** | 稳健，抗噪声 | 收入/成本预测 |
| **梯度提升回归** | 高精度 | 复杂预测场景 |
| **SVR** | 对异常值鲁棒 | 含有离群值的财务预测 |

#### 关键认知
- 回归用于**预测连续值**：预测收入金额、成本金额、坏账金额
- 评估指标：MAE/MSE/RMSE + R²（解释方差比例）
- 交叉验证（10-Fold CV）是评估回归模型稳健性的标准方法
- 实际 vs 预测散点图是诊断模型偏差的最直观方式

### 模块六：无监督学习

**来源**：`unsupervised` / `unsupervised-more` / `embedding_showcase`

#### 掌握要点

| 技术 | 核心方法 | 审计应用 |
|---|---|---|
| **K-Means 聚类** | 将数据分为 K 组，组内相似、组间差异 | 客户分群、供应商分类、交易模式识别 |
| **DBSCAN** | 基于密度，自动发现任意形状簇 + 噪声点 | 地理分布异常、离群交易 |
| **GMM（高斯混合）** | 软聚类，每个点属于各类的概率 | 模糊边界的交易分类 |
| **Isolation Forest** | 随机分割数据，快速隔离异常点 | **异常检测首选**：速度快，无需标签 |
| **LOF（局部离群因子）** | 比较局部密度，发现密度异常 | 局部异常检测（某区域内的异常） |
| **One-Class SVM** | 学习"正常"边界，标记边界外的点 | 已知正常模式，检测偏离 |
| **PCA（主成分分析）** | 降维，保留最大方差方向 | 高维数据可视化、特征去噪 |
| **肘部法则** | inertia vs K 曲线，找拐点 | 确定最佳聚类数 |

#### 关键认知
- 无监督学习在审计中的价值**被严重低估**——大部分审计场景没有标注数据
- Isolation Forest 是审计异常检测的最佳起点：无需标签、速度快、可解释
- PCA 降维后可视化是理解高维数据结构的最佳方式（如把 50 维数据投影到 2D 观察异常分布）
- 聚类 + 异常检测的组合使用最为有效：先聚类找模式，再在各类内检测异常

### 模块七：文本挖掘与 NLP

**来源**：`text mining general` / `sentimental_analysis_cnsenti` / `sentimental assignment` / `API_expanded_teaching` / `API_match_teaching_assignment`

#### 掌握要点

| 技术 | 核心方法 | 审计应用 |
|---|---|---|
| **分词** | `jieba.lcut()` / `nltk.word_tokenize()` | 中文/英文文本切词 |
| **词频统计** | `Counter()` / TF-IDF向量化 | 关键信息提取、文档摘要 |
| **文本相似度** | Cosine Similarity | 合同条款比对、供应商名称匹配 |
| **情感分析** | 词典法(cnsenti/HowNet) + LLM法 | 年报语调分析、投诉文本情绪 |
| **词云** | `WordCloud` | 快速可视化文本重点 |
| **模糊匹配** | 规则候选缩小 + LLM精确判断 | 企业名称对齐、供应商去重 |
| **LLM 结构化提取** | API调用 + JSON输出约束 | 从非结构化文本提取字段 |
| **LLM 批量分类** | 批量API调用 + 缓存 | 大规模文本分类标注 |

#### 关键认知
- **词典法 vs LLM法**：词典法快速但无法处理否定/转折（"产品质量不错，但售后很差"），LLM 能理解语境但成本高
- **混合策略最优**：规则预处理 → 候选集缩小 → LLM 精判（如企业名称模糊匹配）
- 情感分析在审计中的重要应用：年报 MD&A 管理层语调是否过于乐观（预警信号）、投诉文本批量分析
- LLM 的结构化提取能力（JSON 输出约束）是审计文本处理的核心武器

### 模块八：持续审计与风险预警系统

**来源**：`continuous_audit_risk_warning_course_fixed_executed` + 第 05 章 PPT

#### 掌握要点

| 技术 | 核心内容 | 应用场景 |
|---|---|---|
| **规则引擎设计** | 审计疑点 → 数据规则 → Python 实现 | 自动化风险筛查 |
| **风险评分模型** | 加权求和 → 等级分类(高/中/低) | 风险排序、优先级管理 |
| **审计中间表** | 基础性中间表 + 分析性中间表 | 面向审计分析的数据存储 |
| **分析模型层次** | 系统分析→类别分析→个体分析 | 从总体到具体的三层递进 |
| **多维分析** | 按部门/供应商/月份/员工切分 | 多角度锁定风险 |
| **仪表盘** | Streamlit / Matplotlib 可视化 | 持续监控界面 |
| **结果导出** | Excel 多 Sheet 输出 | 审计工作底稿 |

#### 关键认知
- **审计分析模型三大构建依据**：法律法规 → 业务处理逻辑 → 数据勾稽关系 → 外部数据关联 → 审计人员经验
- 持续审计的本质不是替代人，而是**让机器持续处理大量数据，帮助审计人员更早、更快、更有重点地发现风险线索**
- 审计中间表是数据采集和分析之间的**关键桥梁**——将分散的源数据整合为面向审计目标的规范化数据
- 风险权重应基于历史审计发现频率确定，而非主观判断

### 方案一：零售连锁企业——采购付款持续审计

**场景**：某零售企业月均 5000+ 笔采购付款，涉及 200+ 供应商、30 个门店。

**技术方案**：
1. **数据采集**：从 ERP 系统通过 ODBC 导出采购订单表、付款表、供应商主数据
2. **数据预处理**：合并三表（left join），清洗缺失值，验证金额合计
3. **规则引擎**：
   - 大额付款：单笔 > 门店月均付款额的 95 分位数
   - 重复付款：同一发票号+供应商+金额出现 2 次以上
   - 拆分付款：同一供应商同日多笔 < 审批阈值但合计超额
   - 供应商集中度：单一供应商占比 > 15%
4. **风险评分**：各规则加权 → 高/中/低风险排序
5. **异常检测**：Isolation Forest 对高风险交易二次筛查
6. **可视化仪表盘**：Streamlit 实时展示各门店风险排名、月度趋势

**预期产出**：每月自动生成高风险交易清单（Top 50），审计人员只需核查 1% 的交易，覆盖率 99%。

### 方案二：商业银行——信贷资产质量审计

**场景**：某银行 10 万+ 对公贷款客户，需识别隐藏不良贷款。

**技术方案**：
1. **数据采集**：核心系统 + 信贷管理系统导出贷款台账、还款记录、担保信息
2. **审计中间表**：关联贷款主表、还款明细、担保合同、客户工商信息
3. **系统分析**：五级分类结构分析 + 逾期率趋势分析
4. **类别分析**：
   - 关注类贷款中利息回收率 < 80% 的客户
   - 借新还旧且贷款余额增长的客户
   - 担保人/抵押物与借款人存在关联关系的客户
5. **机器学习**：随机森林分类（训练特征：逾期次数、利息回收率、行业风险、担保方式），预测隐藏不良贷款
6. **聚类分析**：对关注类客户分群，识别"表面正常但特征接近不良"的群体

**预期产出**：精准锁定隐藏不良贷款客户名单，替代人工逐户排查。

### 方案三：上市公司——财务报告舞弊识别

**场景**：基于证监会处罚历史数据 + 上市公司财务指标，构建舞弊预警模型。

**技术方案**：
1. **数据整合**：财务指标数据库 + 证监会处罚书文本 + 工商变更记录
2. **文本挖掘**：LLM 从处罚书提取舞弊类型、涉及科目、第三方配合信息（即智能财务前沿作业的核心）
3. **特征工程**：30+ 财务指标（盈利、偿债、营运、现金流）+ 文本特征（年报语调、关键词频率）
4. **不平衡处理**：SMOTE 过采样（舞弊样本通常 < 5%）
5. **模型训练**：随机森林 + XGBoost 对比，PR 曲线评估
6. **特征重要性**：识别舞弊最强信号（如：收入增长与现金流背离、存货周转异常下降）

**预期产出**：上市公司舞弊风险评分排名，辅助审计资源配置。

### 方案四：政府部门——专项资金使用审计

**场景**：某省科技专项资金年度拨付 5000+ 笔，需排查不合规使用。

**技术方案**：
1. **数据采集**：财政拨款系统 + 项目申报材料 + 企业工商数据
2. **企业名称模糊匹配**：拨款名单 vs 工商注册名单（处理简称/曾用名/错别字）
3. **规则引擎**：
   - 同一法人控制的多家企业同时申报同类项目
   - 项目周期结束但资金使用率 < 30%
   - 项目申报金额与历史经营规模严重不匹配
4. **API 验证**：批量调用企业信用 API 核查企业经营状态（是否注销/异常）
5. **聚类分析**：对申报项目按行业+金额聚类，识别异常集群
6. **结果输出**：Excel 多 Sheet（高风险项目清单、企业关联网络、资金使用分析）

### 方案五：电商平台——交易欺诈实时检测

**场景**：日均 100 万+ 交易，需实时识别刷单、虚假交易、套现行为。

**技术方案**：
1. **实时数据管道**：Kafka/Flume 采集交易流水
2. **特征实时计算**：用户历史交易频率、金额分布、设备指纹、IP 地址
3. **规则层**（毫秒级）：单日交易次数 > 阈值、同一设备多账号操作
4. **模型层**（秒级）：Isolation Forest 异常检测 + LightGBM 分类
5. **文本分析**：用户评价情感分析（虚假好评通常语气模板化）
6. **聚类**：对可疑用户按行为模式分群，发现协同作弊团伙

## 第三部分：技术掌握优先级建议

基于审计工作实际需求，推荐学习优先级：

| 优先级 | 技术 | 理由 |
|---|---|---|
| **P0 必学** | Pandas 数据处理 + SQL | 80% 的审计数据分析工作只需要这两样 |
| **P1 核心** | 持续审计规则引擎 + 可视化 | 可直接应用到日常审计项目 |
| **P1 核心** | 监督学习（随机森林） | 分类和回归解决核心审计判断问题 |
| **P2 进阶** | 无监督异常检测 | 发现未知风险模式 |
| **P2 进阶** | 不平衡数据处理 | 审计天然面对不平衡数据 |
| **P3 高级** | 文本挖掘 + LLM API | 处理非结构化审计证据 |
